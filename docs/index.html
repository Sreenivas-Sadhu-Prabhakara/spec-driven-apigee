FIG.01 / THE CORE IDEA

Separate deterministic generation from judgment.

Templates and codegen produce the skeleton — reproducible, reviewable, diffable. The agent supplies the decisions a template can't make: which policies, what test scenarios, the security posture. Knowing which is which is the whole discipline — so this entire page colour-codes it.

◇

Deterministic · Machine

Reproducible across runs. Two runs of the same spec produce byte-identical bundles, so diffs are meaningful and reviews aren't theatre.

The decisions a template cannot encode. Expressed through the spec or template so they're still reproducible — never as a one-off hand-edit to the bundle.

Scenario

Meridian Bank must expose Open Banking accounts to certified fintechs.

Under PSD2-style open-banking rules, Meridian Bank must let certified third-party providers (TPPs) read a customer's consented account, balance, and transaction data — securely, rate-limited per partner, fully audited, with no customer PII leaking through error responses. It is a brand-new API with a hard compliance deadline. The team goes contract-first: write the OpenAPI spec, let the pipeline render and prove the proxy.

FIG.04 / THE CLAUDE CODE SETUP

The harness that makes this hold up in production.

The discipline isn't enforced by good intentions — it's enforced by configuration. One always-true context file, one skill that owns the pipeline, two read-only subagents for a clean second opinion, and hooks that enforce without thinking.

▤
Always-true context

CLAUDE.md the golden rule

Short and always-true: verified commands, conventions, and the rule that prevents the #1 failure mode. If a line wouldn't cause a mistake when removed, it's cut — long files get ignored.

CLAUDE.md — the golden rule
# The golden rule (do not violate)
1. Never hand-edit a generated bundle under apiproxies/.
   To change a proxy, change the spec or the template
   and re-render. A hook blocks manual edits.
2. Never put secrets in specs, bundles, or git.
3. Generation is deterministic; you supply judgment.
⛭
One skill owns the pipeline

scaffold-proxy .claude/skills/

So the workflow runs identically every time instead of being re-prompted. Lint → render → policies → tests → validate → drift, stopping and reporting on any failure. It hands off to the reviewer before declaring done.

SKILL.md — scaffold-proxy
1. Lint the spec first — invalid contract never generates
2. Render bundle deterministically — shared template
3. Layer judgment policies — via template / x-
4. Generate apickli tests — per operation
5. Validate on mock / TEST — self-correct in spec
6. Drift check — re-render + diff
→ hand to policy-reviewer · do NOT deploy
◎
Read-only subagents

proxy-explorer + policy-reviewer

Subagents keep the main context clean and add a second opinion. A cheap haiku explorer maps the spec in plan mode; a judgment-grade sonnet reviewer audits security before deploy. Both get only Read, Grep, Glob.

two read-only agents
proxy-explorer   model: haiku   plan-mode map
  reads spec + template, returns a tight plan
policy-reviewer  model: sonnet  pre-deploy audit
  findings by severity → SAFE / NEEDS FIXES
tools: Read, Grep, Glob   (no write, ever)
⊘
Hooks enforce, they don't think

guard-generated + deploy-gate PreToolUse hooks

Block manual bundle edits at write time — a category error worth stopping instantly. Run the heavier governance gate at deploy time so you don't break the agent's reasoning mid-edit. Exit 2 blocks the call and feeds the reason back to Claude.

guard-generated.sh
# PreToolUse · Edit|Write|MultiEdit
case "$path" in
  */apiproxies/*)
    echo "BLOCKED: generated bundle." >&2
    echo "Change the spec or template." >&2
    exit 2 # block + tell Claude why
esac
